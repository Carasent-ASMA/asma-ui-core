# UI Component Story Standard

This document defines what a **valid Storybook story** is in this repository.

Stories are not demos.
Stories are executable specifications.

A component is considered reliable only when its stories prove its behavior.

---

## Story Categories (MANDATORY STRUCTURE)

Every component must implement stories in these layers.

### 1) Documentation Stories (usage examples)

Purpose: teach developers how to use the component correctly.

Rules:

- No `play()` functions
- No assertions
- Only realistic usage
- Should answer: _how do I use this component?_

Example:

```tsx
export const Default: Story = {
    args: { children: 'Submit' },
}

export const WithIcon: Story = {
    args: { children: 'Filter', startIcon: <FilterIcon /> },
}
```

---

### 2) Behavioral Stories (SAFETY NET — REQUIRED)

Purpose: guarantee UX correctness and prevent regressions.

Rules:

- Must contain `play()`
- Must contain assertions
- Must simulate real user actions only (`userEvent`)
- No implementation details (no class selectors unless unavoidable)

Example:

```tsx
export const ClickFires: Story = {
    args: { children: 'Submit', onClick: fn() },
    play: async ({ canvas, userEvent, args }) => {
        const btn = canvas.getByRole('button', { name: /submit/i })
        await userEvent.click(btn)
        expect(args.onClick).toHaveBeenCalledTimes(1)
    },
}
```

---

### 3) Accessibility Stories (REQUIRED)

Purpose: prevent invisible UI & keyboard traps

You must test:

- keyboard activation
- accessible name
- disabled semantics
- focus behavior

Example:

```tsx
export const KeyboardAccessible: Story = {
    args: { children: 'Submit' },
    play: async ({ canvas, userEvent }) => {
        const btn = canvas.getByRole('button', { name: /submit/i })
        btn.focus()
        await userEvent.keyboard('{Enter}')
        await userEvent.keyboard(' ')
    },
}
```

---

### 4) Visual Gallery (OPTIONAL BUT RECOMMENDED)

Purpose: designers & QA visual reference

Rules:

- NO play function
- NO assertions
- Exhaustive combinations allowed
- Pure rendering only

Example:

```tsx
export const Gallery = () => (
    <Stack gap={2}>
        <Button variant='contained'>Contained</Button>
        <Button variant='outlined'>Outlined</Button>
        <Button variant='text'>Text</Button>
    </Stack>
)
```

---

## What NOT to do

❌ giant story rendering 50 variants with logic\
❌ stories testing implementation details\
❌ snapshot testing components blindly\
❌ chasing 100% coverage\
❌ using stories as playground experiments

---

## Testing Philosophy

We test **user guarantees**, not code paths.

A correct test answers:

> If this breaks in production — will CI fail?

If the answer is no → the story is useless.

---

## Component Risk Checklist

When writing stories, first identify what kind of risk the component introduces.

### Input components

- Typing works correctly
- Value updates propagate
- Clearing resets state
- Validation feedback appears

### Action components (buttons, toggles)

- Click triggers exactly once
- Disabled state blocks interaction
- Keyboard activation works (Enter + Space)

### Navigation components (tabs, menus)

- Arrow key movement
- Focus management
- Selection state updates

### Overlay components (dropdowns, modals, popovers)

- Opens and closes correctly
- Escape closes
- Outside click closes
- Focus is trapped and restored

### Data display components

- Renders correct content
- Handles empty state
- Handles long content / overflow

---

## Naming Convention

Use predictable story names so developers immediately know intent.

#### Default — canonical usage example

#### Disabled — inert state

#### Keyboard — accessibility behavior

#### Interaction — user workflow simulation

#### Error — validation or failure state

#### Gallery — visual reference only

---

## Minimum Required Per Component

- 1 documentation story
- 1 interaction story
- 1 accessibility story

Without these — the component is incomplete.

---

## Goal

Storybook is our UI specification engine.

A refactor is safe only when all behavioral stories pass.
